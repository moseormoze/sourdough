# Feature: מעקב סטארטר (Starter Tracker)

## Problem
היום המשתמש חייב לזכור בעצמו מתי האכיל את הסטארטר וכמה — אין שום תיעוד. אין דרך לחזור אחורה ולראות היסטוריית האכלות, ואין מקום אחד לתעד את זה חוץ מהזיכרון. זה בעייתי במיוחד כי סטארטר דורש טיפול סדיר, ופספוס תיעוד אומר שהמשתמש לא יודע להסתמך על שום דבר כשהוא מתכנן את הבייק הבא.

## User Story
As a home baker maintaining a live סטארטר, I want to log each feeding (when + how much) and see my feeding history, so that I never lose track of how I've been caring for it — regardless of which device I check from.

## Scope — What's In
- מסך/סקשן עצמאי **"מעקב סטארטר"**, נגיש מהבית, נפרד לגמרי מזרימת בייק פעיל
- טופס הזנת האכלה:
  - יחס (סטארטר:קמח:מים) — **חובה**
  - פירוט גרמים לכל רכיב — **אופציונלי**
  - תאריך — **חובה**, ברירת מחדל היום
  - שעה — **אופציונלי**, ברירת מחדל "עכשיו", ניתנת לעריכה (הזנה למפרע)
- רשימת היסטוריה של כל ההאכלות, מהחדש לישן
- עריכה ומחיקה של רשומה קיימת (מחיקה עם אישור — לא בלחיצה אחת)
- **Persistence: Supabase**, ללא מערכת auth חדשה — כל רשומה משויכת לאימייל הקיים מ-`lib/identity/storage.ts` (feature 21)
- מצבי טעינה/שגיאה/ריק בסיסיים לרשימת ההיסטוריה (יש תלות ברשת אמיתית לראשונה בפרויקט)

## Out of Scope
- טיפ/הנחיה לכמות מומלצת להשאיר בצד — v2
- ענף החלטה "מתכנן לאפות בקרוב?" שמשפיע על כמות/מיקום האכלה (מקרר מול חוץ) — v2
- תזכורות / push notifications להאכלה הבאה — v2, תלוי בתשתית web push שעדיין לא קיימת
- חיבור הלוג לחישובי התזמון הקיימים (`13-starter-peak-explainer`, שלב 0 מ-`05-starter-as-stage-zero`) — נשאר נפרד
- יצירת סטארטר מאפס / החייאת סטארטר שלא שרד — כבר ב-backlog
- **אימות זהות אמיתי** — האימייל משמש כמזהה בלבד, לא כבעלות מאומתת (מקובל למשתמש יחיד ב-beta; ידרוש auth אמיתי אם המוצר יגדל)
- תמיכת offline / פתרון קונפליקטים בסנכרון — מעבר למצב טעינה/שגיאה בסיסי

## ⚠️ חריגה מהחלטה קודמת
ה-brief של feature 21 הגדיר במפורש **"Supabase Auth / דאטה בצד שרת — נשאר ב-Phase 4"** כ-Out of Scope. הפיצ'ר הזה מביא Supabase פנימה מוקדם מהמתוכנן — **בלי auth**, רק לצורך שמירת רשומות ההאכלה. `context/tech-stack.md` (נועל "בלי backend" ב-MVP) ו-`context/launch-plan.md` צריכים עדכון שישקף את זה לפני/במהלך שלב Tech Lead.

## Acceptance Criteria
- [ ] מסך "מעקב סטארטר" נגיש מהבית ולא תלוי בבייק פעיל
- [ ] לא ניתן לשמור רשומה בלי יחס (סטארטר:קמח:מים) ותאריך; גרמים ושעה אופציונליים
- [ ] שמירה כותבת רשומה חדשה ל-Supabase, משויכת לאימייל הקיים מ-`lib/identity/storage.ts`
- [ ] רשימת היסטוריה מציגה רק רשומות של האימייל הנוכחי, מסודרות מהחדש לישן
- [ ] ניתן לערוך רשומה קיימת ולשמור שינוי
- [ ] ניתן למחוק רשומה — עם אישור מפורש, לא במחיקה בלחיצה אחת
- [ ] כל query מסונן לפי אימייל — אין דליפה של רשומות בין זהויות
- [ ] מצב טעינה מוצג בזמן קריאה מ-Supabase; שגיאת רשת מציגה הודעה + אפשרות "נסה שוב"
- [ ] מסך ריק (אין רשומות עדיין) מציג הנחיה ידידותית להתחיל להזין
- [ ] עברית/RTL תקין — logical properties בלבד, טאצ'-טרגטים ≥44px

## Dependencies
- Depends on: שכבת הזהות מ-feature 21 (`lib/identity/storage.ts`) קיימת ופעילה; הקמת פרויקט Supabase חדש (env vars, client, טבלה) — תשתית שלא קיימת עדיין בפרויקט
- Blocks: אין פיצ'ר תלוי ישירות, אבל זו הכניסה הטכנית הראשונה של Supabase לפרויקט — משפיעה על תזמון Phase 3/4 של launch-plan
