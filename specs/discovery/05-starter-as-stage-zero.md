# Discovery: Starter כשלב 0 בבייק

## הרעיון

כרגע ה-app מניח שכשהיוזר לוחץ "התחל בייק" — הסטארטר כבר מוכן. הגייט החדש (feature 06) מוסיף שאלה: "הסטארטר בשיא?". אם התשובה "לא" — המסך מציג תכנון עם שעת האכלה מומלצת, אבל הבייק לא מתחיל. היוזר יוצא, מאכיל את הסטארטר לבד, ואחר כך חוזר לבחור מתכון.

**הרעיון לשינוי:** במקום שהאכלת הסטארטר תישאר "מחוץ ל-app", היא הופכת לשלב 0 בתוך הבייק עצמו. היוזר בוחר "לא מוכן", ממשיך לבחור מתכון — ושלב האכלת הסטארטר מתווסף אוטומטית לפני שלב 1. הבייק נשמר במצב "ממתין לסטארטר" עד שהיוזר מאשר שהאכיל.

## מה זה פותר

- **זיכרון**: כרגע היוזר חייב לזכור בעצמו מתי לאכיל ומה לחפש. ה-app לא זוכר בשבילו.
- **רצף**: האופה עוזב את ה-app ועלול לשכוח לחזור, או לבצע את ה-gate פעם שנייה.
- **זמנים**: האכלת הסטארטר משפיעה ישירות על כל שאר הזמנים בבייק. אם זה שלב 0, הזמנים יכולים להיות מדויקים.
- **חוויה**: "הבייק התחיל" תחושה, גם כשאתה עדיין ממתין לסטארטר.

## שאלות פתוחות

1. **Data model**: `activeBake` כרגע מניח `currentStage >= 1`. איך מייצגים "שלב 0"? `currentStage: 0`? שדה נפרד `starterStatus: "pending" | "ready"`?

2. **הגדרת "מוכן"**: מי מחליט שהסטארטר מוכן? היוזר לוחץ "הסטארטר מוכן, ממשיכים"? או שיש טיימר שמציג "אמור להיות מוכן בעוד X"?

3. **Gate flow**: אם הסטארטר מוכן → flow ישיר לבחירת מתכון (ללא שלב 0). אם לא מוכן → בחירת מתכון → שלב 0 נוסף. כלומר ה-gate עדיין קיים, רק התוצאה של "לא" שונה.

4. **ביטול / abandon**: אם היוזר מחליט לא לאפות לאחר שפתח בייק עם שלב 0 — מה קורה? abandon רגיל?

5. **Push notifications**: שלב 0 הוא המקום הטבעי ביותר לשלוח reminder "הסטארטר אמור להיות מוכן עכשיו — בדוק". זה מחייר integreation עם web push. בסקופ?

6. **מה עם בייק פעיל קיים?**: אם יש בייק פעיל ב-"שלב 0" וה-app נסגר ונפתח — מה המסך שמוצג?

## קשרים לפיצ׳רים קיימים

- Feature 06 (starter-readiness-gate): הגייט הנוכחי הוא ה-entry point. שלב 0 הוא evolution שלו.
- Feature 02 (bake-session-shell): המבנה של `activeBake` ישתנה.
- Feature 03 (bake-stages): ה-stage screen יצטרך להציג שלב 0 בצורה שונה משאר השלבים.
